## 引言

PI Agent（Pi）是一个轻量级的**终端编码助手**，由 Alejandro AO 开发，完全开源。与 Claude Code、Codex CLI 等大型编码助手不同，Pi 的设计哲学是**极简 + 可理解**：

- 核心代码量小，架构清晰，适合学习
- 不依赖复杂的框架，从零构建完整的 Agent 系统
- 作为一个教育项目，Pi 展示了编码 Agent 应有的所有核心组件

> "Pi 不是一个黑盒产品，而是一个可理解的架构参考。它的代码量足够小，让你能理解每一行代码的作用；同时它包含了真实 Agent 所需的所有关键组件。"

本文基于一份深度架构解析 PDF，全面拆解 PI Agent 的八大核心模块。

---

## 一、整体架构概览

Pi 由**两个独立层**组成，这是其架构的核心设计决策：

| 层 | 职责 | 技术栈 |
|----|------|--------|
| **Pi Interactive（TUI 层）** | 聊天界面、命令处理、会话管理、技能接口 | Textual 终端 UI |
| **Agent Core（核心层）** | Agent 循环、工具执行、模型调用 | 纯 TypeScript |

**为什么要分层？**

- **关注点分离**：核心层只关心 Agent 循环和工具执行，不关心用户界面
- **可复用性**：同一个 Agent Core 可以被 TUI、CLI、SDK、HTTP API 等多种方式调用
- **可测试性**：核心层可以独立测试，不需要模拟终端界面

### Pi 的核心概念清单

| 概念 | 说明 |
|------|------|
| **Agent Core** | 核心 Agent 循环引擎：上下文构建 → 模型调用 → 响应解析 → 工具执行 |
| **TUI** | 终端用户界面，负责聊天交互、命令处理、会话管理 |
| **Context Init** | 每次模型调用前构建上下文的流程 |
| **Tool System** | 模型通过函数调用与操作系统交互的机制 |
| **Session** | 跨轮次的对话状态持久化 |
| **System Prompt** | 分层的提示词体系（基础 / 项目 / 技能 / 用户） |
| **Extension** | 插件式扩展机制，可添加工具、命令、自定义行为 |
| **Compaction** | 上下文压缩机制，突破长度限制 |
| **Skill** | 可复用的工作流指令，替代人肉记忆复杂流程 |

## 二、Agent Core：核心 Agent 循环

### 2.1 核心循环流程

Agent Core 的核心是一个循环（Loop），每次交互的执行流程完全一致：

```
初始化上下文
    ↓
调用 LLM ←──────────┐
    ↓                │
解析响应              │
  ├── tool_calls → 执行工具 → 追加结果到对话 ──┘
  └── final_answer → 返回答案
```

**详细的每一步：**

1. **初始化上下文（Context Initialization）**：构建发送给模型的消息列表，包括系统提示词、项目指令、可用工具描述、会话历史、用户消息、技能/扩展指令
2. **调用 LLM**：将构建好的上下文发送给语言模型，等待响应
3. **解析响应（Parse Response）**：判断模型的输出是最终答案（Final Answer）还是工具调用（Tool Calls）
4. **执行工具（Execute Tools）**：如果是工具调用，执行对应的工具函数，获取结果
5. **追加结果**：将工具执行结果追加到对话历史中，返回第 2 步继续循环
6. **返回答案**：如果是最终答案，返回给用户，结束本轮交互

> **核心洞察**："有趣的部分不在于有一个循环，而在于在调用模型之前放入了什么到上下文中，以及工具结果如何被反馈到对话中。"——Alejandro AO

### 2.2 上下文构建

当一个 Session 开始时，Pi 构建的上下文包含以下层次：

```
基础系统提示词（Base System Prompt）
    + 项目特定指令（Project Instructions）
    + 可用工具描述（Tool Schemas）
    + 技能/扩展指令（Skills）
    + 会话历史（Session History）
    + 用户当前消息
```

Pi 的基础系统提示词故意保持小巧（small），这让它的行为容易理解和定制。这也是 Pi 成为优秀教育项目的原因之一——架构足够紧凑可以完整理解，但包含了真实编码 Agent 所需的所有核心组件。

### 2.3 会话与对话状态（Session）

Session 在多次交互之间保持连续性，它存储：

- **对话历史（Conversation History）**：完整的消息列表
- **工具执行结果**：每次工具调用的输入和输出
- **必要状态**：继续任务所需的所有信息

> 没有 Session，每一次交互都是孤立的。Agent 无法记住之前读了什么文件、做了什么修改、遇到了什么错误。

## 三、工具系统（Tool System）

### 3.1 工具的桥梁作用

工具是模型与环境之间的桥梁。语言模型本身无法直接读取文件、执行命令或编辑代码——工具赋予了它这些能力：

```
语言模型 → tool_call → 工具系统 → 系统调用 → 操作系统
语言模型 ← result ← 工具系统 ← 输出 ← 操作系统
```

### 3.2 典型工具列表

Pi 作为编码助手，提供了以下核心工具：

| 工具 | 类型 | 功能 |
|------|------|------|
| `read_file` | 文件操作 | 读取指定文件内容到上下文 |
| `write_file` | 文件操作 | 写入或覆盖指定文件 |
| `edit_file` / `patch` | 文件操作 | 精确的查找替换编辑 |
| `run_command` | Shell 执行 | 运行任意 Shell 命令 |
| `search_files` / `grep` | 搜索 | 在代码库中搜索内容和文件名 |
| `list_directory` | 文件系统 | 列出目录内容 |

### 3.3 工具 Schema

每个工具在系统提示词中都有一段描述（Schema），告诉模型：工具名、描述、输入参数（必填/可选）、输出格式。模型根据这些描述决定何时调用什么工具以及传入什么参数。**描述越精确，模型的行为越可靠**——这是 Prompt Engineering 在 Agent 系统中的核心应用。

### 3.4 工具执行流程

```
解析 tool_call → 验证参数 → 执行工具函数 → 格式化结果 → 追加到对话
```

## 四、分层提示词系统

Pi 的提示词系统采用**分层叠加**的方式，每一层在前一层的基础上增加：

| 层级 | 内容 | 作用 |
|------|------|------|
| 第 1 层 | **Base Prompt** | Pi 的核心人格和基础行为规则，保持小巧 |
| 第 2 层 | **Global Instructions** | 用户级别的全局配置，如默认工具链 |
| 第 3 层 | **Project Instructions** | 项目级别指令，如"用 pytest 测试" |
| 第 4 层 | **Skills** | 可复用的工作流指令 |
| 第 5 层 | **User Request** | 当前用户的具体问题或任务 |

这种分层让 Pi 在保持通用性的同时，可以灵活适配不同的代码库。项目 A 可能用 uv 管理 Python，项目 B 可能用 make——通过 Project Instructions 层，Pi 自动调整行为而不需要修改基础代码。

## 五、扩展机制（Extensions）

Extension 是 Pi 的插件系统，允许在不修改核心代码的前提下添加功能。Extension 可以贡献：

- **额外的工具**：如数据库查询工具、API 调用工具
- **自定义命令**：如 `/deploy`、`/test`
- **修改提示词**：向系统提示词追加内容
- **项目特定集成**：按需加载

> "保持核心循环简洁，然后为所有可选的或可定制的内容创建扩展点。"——Keep the core loop small, then create extension points for everything optional.

### Extension 的加载时机

```
Session 开始 → 发现可用 Extensions → 加载配置 → 注册工具/命令/提示词
```

## 六、Pi Interactive：终端 UI 层

Pi Interactive 是终端用户界面层，为 Agent Core 提供可交互的包装器：

| 功能 | 说明 |
|------|------|
| 聊天界面 | 输入框、流式输出、Markdown 渲染 |
| 命令系统 | `/help`、`/sessions`、`/compact` 等命令 |
| 会话管理 | 创建、切换、删除、重命名会话 |
| 压缩控制 | 手动或自动触发上下文压缩 |
| 技能接口 | 加载、查看、应用技能 |
| 状态显示 | Token 用量、延迟、当前会话信息 |

> **核心区分**：TUI 不是 Agent 本身——它是围绕 Agent Core 的一个用户界面。这种分离确保 Agent 运行时（Core）专注于循环和逻辑，TUI 专注于用户体验。

## 七、上下文压缩（Compaction）

### 7.1 为什么需要 Compaction

长时间的会话会面临 LLM 上下文窗口的限制。随着对话增长，早期的重要信息可能会被截断。Compaction 的作用是**压缩或总结较早的对话状态**，保留关键信息的同时保持在窗口内。

```
压缩前：完整的对话历史（可能包含噪音）
  → 触发 Compaction →
压缩后：精简的摘要 + 关键决策 + 已改文件 + 当前阻塞点
```

### 7.2 Compaction 保留的信息

好的 Compaction 会保存：

- **用户的目标**：最初要解决的问题
- **重要决策**：架构选择、技术方案确定
- **已修改的文件**：改了什么、为什么改
- **当前的阻塞点**：卡在什么问题上
- **相关的命令和结果**：关键执行输出

### 7.3 Compaction vs 简单截断

| 维度 | Compaction | 简单截断（Truncation） |
|------|-----------|----------------------|
| 信息保留 | 选择性保留关键信息 | 直接丢掉最早的消息 |
| 语义完整性 | LLM 总结，保持连贯 | 可能中断正在进行的任务 |
| Token 效率 | 高（压缩比通常 5-10x） | 低 |
| 实现复杂度 | 中等（需额外 LLM 调用） | 极低 |

## 八、Skill 技能系统

### 8.1 什么是 Skill

Skill 是**可复用的工作流指令**——它不是给人看的文档，而是给 Agent 看的操作指令。

> Skill = 一段结构化的提示词文本，描述了如何完成一个特定的多步骤工作流。Agent 在加载 Skill 后，将其内容注入到系统提示词中，从而在执行相关任务时遵循预设的流程。

### 8.2 Skill 示例

```markdown
# 视频发布工作流
1. 制作原始剪辑
2. 使用工具生成字幕（Caption）
3. 添加叠加元素（Overlays）
4. 渲染最终视频
5. 上传到发布平台
6. 撰写博客文章
7. 创建社交媒体帖子
8. 在各平台同步发布
```

### 8.3 Skills vs Extensions 的区别

| 维度 | Skills | Extensions |
|------|--------|------------|
| 本质 | 提示词文本 | 可执行代码 |
| 影响范围 | 修改 Agent 的指令和行为 | 添加工具、命令、集成 |
| 是否需要编程 | 不需要（纯文本） | 需要（代码实现） |
| 共享方式 | 文件共享 / Skill 市场 | 包管理器 / 插件市场 |
| 复杂度 | 低 | 中到高 |

## 九、Skill Registry 深度讨论

### 9.1 核心问题

在 PI Agent 的开源社区中，有一场关于 Skill Registry 的有趣讨论，其核心问题是：

> 既然使用 skill 时最终需要通过 `/skill` 指定 location，那为什么还要提前把所有 skills 的 `name + description` 放进 system prompt？

**一句话结论**：`name + description` 让模型知道"有哪些技能、什么时候该用"；`location` 让运行时知道"去哪里读取完整内容"。前者解决**发现与路由**，后者解决**加载与执行**。

### 9.2 三层设计

| 层级 | 内容 | 作用 |
|------|------|------|
| **Skill Index** | name + description | 给模型做选择，相当于技能菜单 |
| **Skill Pointer** | location | 告诉系统去哪找完整技能 |
| **Skill Body** | SKILL.md + 相关文件 | 真正指导模型怎么做任务 |

### 9.3 为什么不是"把 skill 内容直接塞进 prompt"？

真实的 skill 是一个**可延迟加载的技能包**，可能包括：SKILL.md 完整操作说明、示例文件、脚本、模板、工具调用规范、领域工作流、约束条件。如果一开始就把所有完整内容塞进 system prompt，会浪费大量 context。

这正是 **渐进式披露（Progressive Disclosure）** 原则的体现——PI Agent 官方文档明确指出："only descriptions are always in context; full instructions load on-demand."

### 9.4 三种交互模式

| 模式 | 触发方式 | 适用场景 |
|------|---------|----------|
| **手动模式** | 用户 `/skill:name` | 交互明确、用户控制强的产品 |
| **自动模式** | 模型自主判断 | 真正 agentic 的系统 |
| **混合模式** | 两者兼有 | 最常见，也最合理 |

### 9.5 关键分歧：location 要不要放进 system prompt？

| 方案 | 内容 | 优点 | 缺点 |
|------|------|------|------|
| **A** | 只暴露 name + description | 省上下文、更安全 | 模型知道该用但不知去哪读 |
| **B** | name + desc + location | 简单透明 | 暴露路径、占上下文 |

PI Agent 的实现采用方案 A，同时提供 `disable-model-invocation` 配置项来控制特定 skill 是否对模型可见——这恰恰验证了核心观点：**name-description 的价值取决于系统的交互模式**。

## 十一、Pi 对 Agent 设计的启示

1. **分层架构是基础**：核心层和 UI 层分离让系统更灵活
2. **提示词系统需要分层**：Base + Global + Project + Skill 的叠加模式
3. **工具是 Agent 能力的来源**：工具系统设计的好坏决定了 Agent 的边界
4. **扩展性来自插件化**：Extension 机制让核心保持简洁
5. **Compaction 是长会话的必需品**：没有它，Agent 无法处理复杂任务

### 与其他编码 Agent 的对比

| 维度 | Pi | Claude Code | Codex CLI | Hermes |
|------|-----|-------------|-----------|--------|
| 开源 | ✅ | 部分 | ✅ | ✅ |
| 架构复杂度 | 低 | 高 | 中 | 中 |
| Skill 系统 | ✅ | ✅ | AGENTS.md | ✅ |
| 上下文压缩 | ✅ | Headroom | compact API | Headroom |
| 代码量 | 小 | 大 | 中 | 中 |
